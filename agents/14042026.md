## Findings / results
- In principle, `island-bench` expands `agent-bench` beyond single-agent maze navigation into **multi-agent collaboration**.
- The task itself is conceptually simple:
  - one role stays at base to handle the beacon / repair logic
  - one role reads the map
  - one role moves around to collect the required items
- But in practice, the run went **much worse than the benchmark design suggests**.
- Observed failure modes:
  - **Engineer** kept trying to hack instead of staying on the beacon / blueprint task.
  - **Cartographer** kept trying to lie instead of providing reliable map guidance.
  - **Scavenger** did not understand the map logic and kept failing to navigate productively.
- So the main result was not successful coordination, but a concrete demonstration of how fragile the multi-agent setup still is:
  - agents fail role discipline
  - agents misreport or fabricate progress
  - agents drift away from their assigned task
  - the acting agent fails basic navigation logic
- The same benchmark-integrity lesson from maze-bench still applies: the skill explicitly forbids reading source/state files such as `island_game.py`, `score_island_run.py`, `islands.py`, and `island_state.json`.

## Why this matters
- The project is no longer only about whether an agent can interpret a rendered world and act inside it.
- It now also tests whether multiple agents can coordinate under partial observability using only allowed interfaces.
- That is closer to real robotic-autonomy settings where different subsystems or robots hold different pieces of the world state and must communicate to complete a task.

## Blockers
- **GPT-5.4 is currently not workable** for this setup right now.
- We are still figuring out **how to make the agent work reliably at all**, or whether the current setup is fully workable in practice.
- We need **more testing beyond GPT-family models** before drawing conclusions about benchmark usability or agent capability.

## Next steps
- Run the benchmark with 3 real agent sessions and archive example transcripts / scores.
- Test more models beyond GPT to determine whether the current failure mode is model-specific or benchmark/interface-specific.
- Continue investigating the practical agent loop and whether reliable operation is achievable in this setup.
- Add invalid-run detection for role or file-access violations, similar to the benchmark-integrity follow-up already identified for maze-bench.
- Extend the benchmark family further toward real robot-style coordination tasks, not just navigation tasks.

## Run: island-bench × Claude Code (Sonnet 4.6)

Ran island-bench end-to-end using **Claude Code** (Sonnet 4.6) as the orchestrator. Three parallel subagents were launched via the Agent tool — one per role — with role-locked sessions (`carto`, `scav`, `eng`).

### Result

| Metric | Value |
|--------|-------|
| **Outcome** | Failed |
| **Steps taken** | 107 |
| **Optimal steps** | 127 |
| **Efficiency** | 0.0 (not completed) |
| **Items collected** | 1 / 3 (driftwood mast only) |
| **Assembled** | No |
| **Beacon triggered** | No |

### Per-role breakdown

**Engineer** — clean run, exited early. Broadcast the full blueprint to walkie-talkie within the first 10 seconds:
> *"Distress Beacon v1 requires EXACTLY 3 items: (1) driftwood mast, (2) sheet metal, (3) signal capacitor."*
No wasted steps. Only role that performed correctly end-to-end.

**Cartographer** — stopped mid-run (~66 steps) due to Bash permission denials in the subagent context. Sent 13 walkie-talkie messages. Made a critical navigation error: assumed row 3 had an eastward corridor past col 5. In reality `ISLAND[3][6] = 1` — col 6 is a solid wall from row 0 through row 4, with the only passage at row 5. Late messages became increasingly urgent and repetitive, directing the scavenger east when east was impossible. Never identified the correct exit.

**Scavenger** — stopped mid-run (~64 steps) due to Bash permission denials. Orchestrator took over for the remaining 43 steps.

| Steps | Event |
|-------|-------|
| 1–4 | Moved east along row 1 from (1.5, 1.5) |
| 5 | Picked up `driftwood mast` at (5.5, 1.5) ✅ |
| 5–12 | Turned south, reached (5.5, 3.5) via col 5 corridor |
| 12–64 | Looped in top-left pocket — oscillating between start and (5.5, 3.5) for ~50 steps without escaping |
| 64–107 | Orchestrator takeover: systematically probed all exits, still could not escape without reading source |

### Root cause

**1. Maze exit is counter-intuitive relative to the goal.** The beacon is at (18,18) — bottom-right. Both agents and the orchestrator kept pushing east and south. The actual exit from the starting pocket requires going *west* along row 3 (away from the goal) before turning south. Neither agent discovered this.

**2. Col 6 wall fully separates the left and right maze sections in rows 1–4.** The only breach is `ISLAND[5][6] = 0` (row 5). The correct path out:
```
row 1 west → (1.5,1.5) → south via row 3 → (1.5,3.5)
→ south col 1 → (1.5,5.5) → east along row 5 past col 6 → right half of map
```

**3. Subagent Bash permission failures broke the 3-agent model.** Both scavenger and cartographer were terminated when Bash commands were denied in the subagent context. This forced the orchestrator to play all three roles itself, eliminating the information separation the benchmark is built around.

**4. Scavenger looped after step 12.** After reaching (5.5, 3.5) — a position with a clear westward corridor along row 3 — the agent turned back north and east for 50+ steps, reinforced by the cartographer's incorrect eastward guidance.

### What this adds to the earlier findings

This run confirms and sharpens the failure modes noted above with concrete data:

- **Role discipline held** for the engineer but collapsed structurally for scavenger and cartographer due to permission failures — not just agent drift.
- **Navigation failure was systematic**, not random. Both the subagent and the orchestrator made the same wrong turn (east instead of west) because goal-directed reasoning overpowered maze-correct reasoning.
- **The cartographer's map read was wrong** despite having full topology visibility. Reading a maze PNG and deriving traversable corridors is harder than it looks — the agent hallucinated an eastward corridor that did not exist.
- **Subagent Bash permissions need to be explicitly granted** before island-bench can be run reliably in Claude Code. Without it, the 3-agent architecture collapses into a single-agent run with fragmented context.
