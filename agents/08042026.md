## Structure we are building
- `skill-as-bench.md` → **Agent interface/spec** (the agent consumes benchmark instructions as a skill)
- Environment as code → **Executable benchmark world** (current example: `maze_game.py`)
- Outcome verifier → **Objective evaluation** (current example: `score_maze_run.py`)

```text
+---------------------------+
|      skill-as-bench.md    |
|  (agent-facing benchmark) |
+------------+--------------+
             |
             v
+---------------------------+

|   (maze_game.py runtime)  |
+------------+--------------+
             |
             v
+---------------------------+
|      Outcome Verifier     |
|   (score_maze_run.py)     |
+------------+--------------+
             |
             v
+---------------------------+
|   Comparable Bench Result |
| (success, steps, optimal, |
|  efficiency, rating)      |
+---------------------------+
```

## Additional key reasons (5:00 PM)
- A real agent should operate continuously without interruption. If the user has to step in and set things up mid-run, that is already a failure of the agent loop ❌.
- Capabilities like auto-compact already exist, so the harness should remain as simple as possible. The goal is autonomy: continual operation without failure, not a system that depends on manual babysitting.
- Ease of use matters because it directly affects distribution and scale. If the harness is easy to run, more people will use it, which strengthens the benchmark and helps it spread in the same way successful evals like lmarea and SWE-bench do.
- Multi-agent swarms should be operationally trivial — ideally no more than a simple shell-script `for` loop.
